# プロジェクト共通ルール

## 実装前に必ずやること

**実装・テスト作成を始める前に、以下の順序でドキュメントを確認すること。**

1. `PROJECT_MEMO.md` を読み、仕様・タスク進捗・ディレクトリ構成を把握する
2. `tasks.md` を読み、直近の作業状況・未着手タスクを確認する
3. 該当する実装箇所のコードを読み、既存の構成・命名規則を把握する

**確認なしに実装を始めてはならない。**

- 仕様と異なる実装をしてはならない
- 既存の構成・命名規則・技術スタックに従うこと（詳細は `PROJECT_MEMO.md` の「技術スタック」「主要な設計判断」を参照）
- 不明点はユーザーに確認してから進める

## タスク進捗の更新ルール

**実装が完了したら、以下の 2 ファイルを必ず更新すること。**

### `PROJECT_MEMO.md`
- 完了した項目は `- [ ]` → `- [x]` に変更する
- 新しく発生したタスクがあればバックログに追記する

### `tasks.md`
- プロジェクトルートの `tasks.md` に作業ログを追記する
- フォーマット:
  ```
  ## YYYY-MM-DD
  - [完了] やったこと
  - [追加] 新たに発生したタスク
  ```
- `tasks.md` が存在しない場合は新規作成する

## テストに関するルール

**テストを通すためにテストコード（仕様）を変更してはならない。**

テストが失敗したとき、必ず実装コードを修正して通すこと。

### 禁止事項

- `*.test.ts` / `*.spec.ts` ファイルの `describe` / `it` のブロック内容を変えること
- `expect(...).toBe(...)` などの期待値を変えること
- テストケースを削除・スキップ（`it.skip` / `xit`）すること
- `PROJECT_MEMO.md` の「ユニットテスト仕様」セクションに記載された仕様（入力・期待値）を変えること

### テスト失敗時の対応手順

1. エラーメッセージを読み、**どの実装コードが原因か**を特定する
2. 実装コードを修正してテストを通す
3. 他のテストが壊れていないか `npm run test:run` で確認する

### 例外（ユーザー確認が必要な場合）

以下の場合のみ、ユーザーに確認したうえでテストコードを変更してよい。

- テストコード自体にバグがある（期待値の記述ミスなど）
- 仕様変更によってテストを更新する必要がある
